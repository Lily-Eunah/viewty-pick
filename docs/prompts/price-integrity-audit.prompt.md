# Claude Code 작업 프롬프트 — 가격 무결성 audit (실값 대조 · 읽기 전용)

> 목적: "가격을 신뢰할 수 없다"는 문제의 **근본 원인을 제품별로 진단**한다.
> 각 플랫폼의 **실제 현재 값을 직접 확인**해 우리 DB(`price_snapshots`/`current_prices`/공개 뷰)·화면 표시값과 대조하고,
> 어디서 틀어지는지(매칭/세트·다중팩 정규화/올영 수집/판매가) 규명한다. **쓰기 금지, 진단·리포트만.**

---

## 0. 원칙
- **READ-ONLY.** DB·시트 수정 없음. 라이브 조회만(네이버 API·쿠팡 검색 API·올영 via 네이버). 시크릿 비노출. 레이트리밋 준수.
- 실값은 **운영자 관측치 + 에이전트의 라이브 조회** 둘 다로 교차 확인.
- 산출물 = 제품별 근본원인 표 + mock 출처 인벤토리 + 수정 범위 제안.

## 1. 대상 제품 (운영자 신고 + 대표 케이스)
| 제품 | 운영자 관측 실값 / 메모 |
|---|---|
| 랑콤 제니피끄 얼티미트 세럼 | **세트(세럼21ml + 토너100ml 등), 실제 282,200원** — 화면값/단품 오인 의심 |
| 유세린 하이아르론 에피셀린 세럼 | **실제 60,900원** — 화면값 불일치 |
| 비플레인 녹두 라하 토너 | **쿠팡 6개 67,800(개당 11,300) / 네이버 16,800(1개)** — 정렬·개당가 |
| 토리든 다이브인 포맨 올인원 | **올영 가격 미수집 의심** |
| 후시다딘 더마 트러블 징크 선크림 | **5개 세트** — 리스트에 48,160원 단독 표기(팩가/개당가?) |
- 추가로 디렉터파이 TOP·"오늘 가격 좋은 제품"에 뜬 고가 브랜드(아르마니 등)가 실데이터인지 seed인지 함께 확인.

## 2. 제품별 진단 절차
각 제품에 대해:
1. **실값 확보** — 해당 listing의 각 판매처(네이버 공식몰 / 올영 via 네이버 / 쿠팡)에서 **현재 실제 값**을 라이브 조회: 판매가, 정가, **구성(단품 vs 세트/기획/N개입), 총 용량, 수량**, 프로모션, mallName/productId, 링크.
2. **저장값 대조** — `price_snapshots`(최신)·`current_prices`·공개 뷰(`listing_prices_public`)에 저장된 값과 비교.
3. **화면값 대조** — 리스트/상세에 표시되는 값과 비교.
4. **불일치 분류** — 아래 중 무엇인지 규명:
   - **매칭 오류**: 다른 상품/판매처를 매칭(productId/mallName/제목 유사도).
   - **세트/다중팩 미파싱**: `packageExtractor`가 세트(토너+세럼)·N개입·기획을 못 잡아 `total_ml`/`quantity`/개당가가 틀림 → base/effective/unit_price 오류.
   - **올영 미수집**: 네이버에 올영 오퍼가 안 떠 no_offer인데 화면엔 다른 값/누락.
   - **판매가 오류**: 정가/세일가/프로모션 파싱 실패, parse_confidence.
   - **용량 오류**: `products.volume_ml`이 실제와 다름(§1b 용량 감사).

## 3. 정규화/정책 검증 (특히 다중팩·세트)
- 라하 토너·더마 선크림·랑콤 세트로 **2트랙 계산 검증**: `base_unit_price`(1개 조건없이) / `effective_unit_price`(개당) / `unit_price`(ml당) 가 맞게 계산되는지.
- **세트(이종 구성: 토너+세럼)**는 단순 "개당가"로 환산 불가 — `packageExtractor`/normalize가 이걸 어떻게 처리하는지(또는 못 하는지) 확인. 처리 불가면 **비교 제외 후보**.
- **최저가 정렬 기준**(리스트)이 base인지 unit_price인지 확인 — 라하 토너에서 쿠팡(개당 저렴)이 상위로 오는지.
- **가격 없는(null) 제품**이 최저가순에서 어디에 정렬되는지(뒤로 가야 함).

## 4. mock/seed 출처 인벤토리
코드·데이터에서 다음의 **출처를 특정**(어디서 생성/주입되는지), 제거 대상인지 판정:
- **별점·리뷰 개수**(MVP 리뷰 없음 → mock).
- **"어제보다 OO원 저렴" / "OO원 하락" 뱃지**(가격 히스토리 자산 아직 없음 → mock).
- **"20% 인하" 뱃지**(전 제품 동일 20% → mock 정황).
- **고가 데모 브랜드/제품**(랑콤·아르마니·유세린·바이오힐 등)이 시트(실큐레이션)에서 왔는지, seed/mock 스크립트에서 왔는지.
- 진짜 뱃지(디렉터파이/화해/올영베스트)와 구분.

## 5. 크롤링 동작/신선도 점검
- 최근 sync 시각(`listings.last_crawled_at`, `crawl_runs`/`crawl_errors` — crawl_runs 미기록 이슈도 확인), 어떤 제품이 언제 갱신됐는지.
- 매칭 실패/no_offer/ data_error 분포(현재 라이브 카탈로그 기준).

## 6. 산출물 (리포트만, 쓰기 없음)
- **제품별 근본원인 표**: 제품 | 판매처 | 실값 | 저장값 | 화면값 | 불일치 유형 | 원인 추정.
- **mock 출처 인벤토리**: 항목 | 출처(파일/스크립트/테이블) | 제거 권고.
- **정규화/정렬 정책 검증 결과**: 다중팩·세트·null 가격 처리의 실제 동작 vs 의도.
- **수정 범위 제안**: (A) 크롤/정규화/매칭 (B) mock 제거+뱃지 정책 (C) 표시·정렬·placeholder. 각 우선순위.
- worklog `docs/worklog/price-integrity-audit.md`에 기록(읽기 전용 결과).

## 7. 막히면
- 세트(이종 구성) 개당가 산정이 본질적으로 불가하면 그 사실과 "비교 제외 / base만 표기" 옵션 보고.
- 레이트리밋으로 전수 확인이 길면 대상 제품을 줄이고 명시.
- 라이브 조회가 환경상 막히면 추측 말고 "확인 불가 + 사유" 보고.

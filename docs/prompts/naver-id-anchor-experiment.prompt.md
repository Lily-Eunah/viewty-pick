# Claude Code 작업 프롬프트 — 네이버 productId 앵커링 가능성 실험 (READ-ONLY)

> 목적: 검색 API 결과로 **큐레이션 URL의 채널상품번호(channelProductNo)로 앵커링이 되는지** 결정한다.
> 되면 B(매칭)를 **id 앵커링**으로 재설계(변형/세트 오매칭을 근본 해결), 안 되면 제목/변형 매칭 유지.
> **DB·시트·코드(기능) 변경 없음 — 순수 진단.**
>
> 배경: 검색 API의 `productId`는 스마트스토어/브랜드스토어 상품이면 채널상품번호에, 가격비교 카탈로그면 원상품번호에 매핑(네이버 포럼). 즉 검색이 **개별 몰 상품**을 돌려주면 URL의 번호와 일치할 수 있다.

---

## 0. 안전 원칙
- **READ-ONLY**: DB/시트/`crawler:sync`/`sheets:import` 변경 없음. 라이브 검색 API + 링크 리다이렉트 해석(redirect follow, 콘텐츠 스크래핑 아님)만. 레이트리밋 준수, 시크릿 비노출.
- 진단 스크립트(`scripts/ops/naver-id-anchor-experiment.ts`) + 리포트만 커밋 가능.

## 1. 대상 listing
- 네이버 활성 listing 중 **채널상품번호(N) 추출 가능**한 것:
  - `brand.naver.com/{store}/products/{N}` / `smartstore.naver.com/.../products/{N}` → N 직접 추출.
  - `naver.me/...` shortlink → **리다이렉트 해석**해 최종 URL에서 N 추출(가능하면).
  - N 추출 불가(완전 단축/비표준) → "id 없음"으로 분류.

## 2. 각 listing 절차
1. 매처와 동일한 쿼리(brand + 정규화 name)로 검색 API 호출하되 **`display`를 최대(예: 100)** 로 — 개별 몰 상품이 결과에 들어올 확률을 높임.
2. **모든 결과 항목**의 필드 덤프: `productId`, `productType`, `mallName`, `title`, `lprice`, `link`.
3. **앵커링 가능성 판정**:
   - **(a) 직접 일치**: 어떤 항목의 `productId == N` ?
   - **(b) 링크 경유**: 어떤 항목의 `link`를 해석하면 `.../products/{N}` 으로 떨어지는가?
   - **(c) 불가**: N이 어떤 항목의 productId·해석링크에도 없음 → 검색이 개별 몰 상품을 안 줌(어떤 productType들이 왔는지 기록 — 카탈로그 대표 vs 몰).

## 3. 산출물 (리포트만)
- **per-listing 표**: 제품 | URL의 N | 검색 결과 수 | productType 분포 | (a)직접일치? | (b)링크일치? | 판정(직접/링크/불가).
- **hit-rate 요약**: 직접 N% · 링크 N% · 불가 N% (전체 / brand.naver.com만 / shortlink만).
- **productType 분포**: 결과가 주로 카탈로그 대표인지 개별 몰 상품인지 — 불가 사유 설명.
- `docs/worklog/naver-id-anchor-experiment.md`에 기록.

## 4. 결정 가이드 (리포트 말미)
- **직접+링크 hit-rate가 높으면** → **B를 id 앵커링으로 재설계** 권고: 검색 결과에서 N과 일치하는 항목을 채택 → **운영자가 큐레이션한 정확한 SKU** 확보(변형 오매칭·세트 오매칭 동시 해결). 제목 유사도는 보조로만.
- **낮으면**(검색이 개별 몰 상품을 거의 안 줌) → 기존 B(변형 토큰 제목 매칭) 유지. id 앵커링은 가능한 listing에만 부분 적용.
- 링크 경유만 되는 비율이 크면 → 링크 해석 비용/안정성(레이트리밋·차단) 트레이드오프도 같이 보고.

## 5. 제약
- 쓰기 일절 없음. 링크 해석은 **리다이렉트 추적(최종 URL 확인)**까지만, 페이지 콘텐츠 파싱·복제 금지.
- 검색 호출량 보고(네이버 25k/일 내). 대상이 많으면 배치로.
- 기존 매처 함수(검색 호출부)를 재사용해 쿼리 일관성 유지 — 재구현으로 쿼리가 달라지지 않게.

## 6. 막히면
- shortlink 해석이 막히거나 N이 안 나오면 그 listing은 "id 없음"으로 두고 비율에 반영(추측 금지).
- `productId`/`link` 포맷이 예상과 다르면 실제 샘플 응답을 리포트에 첨부하고 판정 보류.
- 라이브 호출이 환경상 막히면 "확인 불가 + 사유" 보고.

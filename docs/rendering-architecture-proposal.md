# 렌더링 아키텍처 개선 제안 — 제품(주간) / 가격(일간) 주기 분리

작성일: 2026-06-30
배경: [performance-analysis.md](performance-analysis.md)에서 "캐시가 dummy라 모든 페이지가 매 요청 풀 렌더"가 느림의 근원으로 확인됨. 운영자가 "제품은 주 1회 업데이트, 가격은 매일 갱신, 추천/최저가는 매번 계산하지 말고 유지"를 제안 → 그 타당성과 구체적 설계를 정리한다.

---

## 0. 한 줄 결론

제안의 방향은 **정확하다.** 핵심 통찰 = **"제품 카탈로그는 주간, 가격은 일간으로 변하니, 비싼 계산(전 상품 매핑·매칭·정렬)을 매 요청이 아니라 데이터가 바뀔 때만 1번 하고 결과를 재사용한다."** 이건 캐시(ISR) + 온디맨드 재생성으로 거의 그대로 구현되며, 필요한 인프라(`/api/revalidate`, 일간 크롤러, `generateStaticParams`)는 **이미 절반 존재**한다.

단, **무한 스크롤(lazy 로딩)만은 우선순위가 낮다** — ~94개 소형 카탈로그에선 과한 복잡도이고, 진짜 병목(서버 측 캐시 부재)을 건드리지 못한다. 같은 목표(빠른 초기 로드)는 캐시 + 경량 페이로드로 더 싸게 달성된다.

| 제안 | 평가 | 채택 |
|---|---|---|
| 캐시 적용 | ✅ 필수 전제 | P0 |
| 최저가·추천을 하루 1번만 계산(매 요청 X) | ✅ 정확 | P0 — ISR + 온디맨드 revalidate |
| SEO 비교페이지를 static 페이지로 | ✅ 정확 | P0 — `generateStaticParams` + ISR |
| 제품 주간 / 가격 일간 분리 | ✅ 정확, 깔끔 | P0 — revalidate 주기로 실현 |
| lazy 로딩 / 무한 스크롤 | ⚠️ 소형 카탈로그엔 과함 | P2(선택) — 더 단순한 대안 권장 |

## 1. 제안을 "데이터 주기"로 번역

제안의 본질은 **무엇이 얼마나 자주 변하는가**에 따라 계산을 분리하는 것이다:

| 데이터 | 변하는 주기 | 누가 바꾸나 | 비싼가? |
|---|---|---|---|
| 제품 목록·카테고리·배지·viewty_score·SEO 페이지 구성 | **주 1회** | 운영자(시트 import) | 매핑·매칭이 비쌈(O(N×L)) |
| 추천 순위·SEO 페이지별 제품 선정 | 제품에 종속 → **주 1회면 충분** | 위에서 파생 | 비쌈 |
| 가격·최저가·할인율·재고 | **매일** | 크롤러(GitHub Actions) | id 조회는 쌈 |
| 사용자별 최근 본/찜 | 실시간 | 브라우저 | 클라이언트(localStorage) |

→ "매 요청 계산"을 "**데이터가 바뀔 때 1번 계산 후 캐시 재사용**"으로 바꾸면, 위 비싼 계산을 주 1회(제품) / 일 1회(가격)로 줄일 수 있다. 이게 제안의 핵심이고 타당하다.

## 2. 아이디어별 분석

### 2.1 캐시 적용 — ✅ 필수 (모든 것의 전제)

다른 모든 아이디어("하루 1번 계산", "static 페이지")는 **저장할 캐시가 있어야** 성립한다. 현재 `incrementalCache: "dummy"` + R2/KV 바인딩 없음이라 캐시가 0이다. 따라서 **R2(인크리멘털 캐시) + KV(태그 캐시) 연결이 0순위 전제.** 이게 되면 `revalidate`와 `revalidatePath`가 비로소 실제로 동작한다.

### 2.2 "최저가·추천을 하루 1번만 계산" / "추천을 일주일 유지" — ✅ 정확

핵심 요구는 **"매 요청마다 계산하지 말 것"** 이다. 이를 만족시키는 가장 깔끔한 캐시 주기는 **일 1회(daily)** 다. 이유:

- 일 1회 재생성하면 **가격(일간)은 매일 신선**해진다 → "가격은 매일 들고와야 한다" 충족.
- 제품은 주간이라 대부분의 날엔 그대로 → **추천/순위는 사실상 한 주 동안 안정**적으로 유지된다(운영자 의도 충족). 굳이 7일 동결할 필요 없이, "요청마다 X, 하루 1번 O"이면 충분.
- 즉 **revalidate 주기 = 가격 갱신 주기(일 1회)** 로 잡으면 "제품 주간/가격 일간"이 자동 정렬된다.

> ⚠️ 주의: 현재 "추천/픽" 일부는 **가격에 의존**한다.
> - `recommended` 정렬 = `byPriceThen(viewtyScore desc)` — `byPriceThen`이 **가격 없는 상품을 맨 뒤로** 보냄 → 오늘 모든 가격이 빠진 제품은 그 주 안에서도 순위가 내려감(이건 바람직한 동작).
> - `officialPicks`(정가 대비 픽) = `discountVsRegular > 0` 필터·정렬 — **완전히 가격 종속(일간)**.
>
> 결론: **"큐레이션·순위 골격은 주간 안정 + 가격/재고 값은 일간 갱신"** 으로 가되, 굳이 추천 선정을 7일 동결하기보다 **일 1회 재계산**이 더 단순하고 정확하다(가격 의존 부분이 매일 자연스럽게 맞춰짐). 진짜로 "선정 자체를 주 동결"하고 싶다면 2.5의 스냅샷 방식이 필요하다.

### 2.3 "SEO 비교페이지를 static 페이지로" — ✅ 정확

지금 `/best/[slug]`는 [generateStaticParams](app/best/%5Bslug%5D/page.tsx)로 빌드 때 프리렌더까지 해두는데, **dummy 캐시 때문에 그 결과가 저장·서빙되지 못해 매번 다시 매핑**한다(게다가 `generateMetadata`+본문에서 2번). 캐시만 연결하면:

- 빌드 시 프리렌더된 정적 결과가 R2 캐시에 저장되고,
- 요청은 그 **정적 결과를 즉시 서빙**(DB·매핑 0),
- 일 1회 revalidate 시점에만 백그라운드 재생성.

→ "static으로 만들어둔 페이지를 보여줘"가 **ISR로 그대로 실현**된다. (완전 정적이 더 좋으면 2.5의 "풀 SSG" 옵션.)

### 2.4 "lazy 로딩 / 무한 스크롤" — ⚠️ 우선순위 낮음, 대안 권장

직관("필요할 때만 가져와라")은 옳지만, **이 프로젝트엔 잘 안 맞는다:**

- **병목은 클라이언트가 아니라 서버다.** 느림의 원인은 "브라우저에 제품이 많아서"가 아니라 "서버가 매 요청 풀-DB 스캔 + O(N×L) 매핑"이기 때문. 무한 스크롤은 클라이언트 페이로드만 줄일 뿐, 캐시 없이는 서버 비용을 못 줄인다. 오히려 스크롤마다 API 요청이 다시 Worker→DB를 때리면 **요청당 부하가 늘 수도** 있다.
- **카탈로그가 작다(~94개).** 무한 스크롤은 수천~수만 항목에서 의미가 있다. 94개는 한 번에 보내도 되는 규모.
- **리스트는 전체를 정렬/필터**(최저가·추천순)해야 해서, 페이지네이션해도 **정렬된 전체 집합이 서버에 먼저 있어야** 한다. 결국 서버 계산은 그대로.
- **SEO 페이지는 전 상품이 HTML/JSON-LD에 있어야** 크롤러가 읽는다 → 무한 스크롤은 **SEO에 해롭다**. `/best/*`엔 절대 적용 금지.

**같은 목표(빠른 초기 로드)를 더 싸게 얻는 대안:**
1. **(권장) 캐시 + 경량 페이로드**: 캐시로 서버 비용 제거 후, 홈의 `allProducts` 전량 전달(C6)을 **카드에 필요한 필드만 추린 경량 배열**로 축소.
2. **서버 컴포넌트로 리스트 렌더** + 상호작용 부분만 작은 클라이언트 컴포넌트 → 하이드레이션 데이터 자체를 줄임.
3. 정말 길이가 길면 **"더 보기"로 이미 받은 배열을 잘라 보여주기**(추가 요청 없음) 또는 DOM 가상화. 무한 스크롤(추가 네트워크 요청)은 마지막 수단.

→ 결론: lazy 로딩은 **캐시·정적화가 끝난 뒤의 선택적 폴리시(P2)**. 지금 0순위 아님.

### 2.5 (선택) 더 강한 옵션 — 스냅샷 / 풀 SSG

ISR 일 1회 재생성조차 무겁다고 느껴지면(94개에선 거의 아님) 두 가지 상위 옵션:

- **스냅샷 테이블/JSON**: 크롤러·주간 잡이 "SEO 페이지별 제품 목록 + 추천 순위 + 최저가"를 **미리 계산해 denormalized 스냅샷**(테이블 1행 또는 R2/KV의 JSON)으로 저장. 페이지는 **키 1개 조회 후 렌더** → 카탈로그 크기와 무관하게 O(1). "선정 자체를 주 동결 + 가격만 일간 덮어쓰기"를 가장 정확히 구현. (구현량 큼)
- **풀 SSG + 재빌드**: 제품 import(주간)·가격 크롤(일간) 후 `next build` + `cf:deploy`를 자동 실행 → 페이지가 **순수 정적 에셋**(Worker 렌더 0). 가장 빠르고 Worker 비용 최저. 단 **가격 매일 반영 = 매일 전체 재빌드/배포**라 운영 부담이 ISR보다 큼.

## 3. 권장 아키텍처 (이미 있는 자산 재사용)

```
[ 주 1회 ] 운영자 제품 시트 import (sheets:import + crawler:sync)
      │
      └─▶ POST /api/revalidate  (전체 핵심 경로 + /best/* 태그)
                                  └─ 제품 골격/추천/SEO 페이지 재생성

[ 매일 ] 크롤러 가격 갱신 (GitHub Actions, 이미 운영 중)
      │   ※ crawler/run.ts:935의 주석 처리된 revalidate 호출을 활성화
      └─▶ POST /api/revalidate
                                  └─ 가격/최저가/할인 값 재생성 (제품 골격은 그대로)

[ 요청시 ] 사용자 접속
      └─▶ R2 인크리멘털 캐시 HIT → 저장된 정적 결과 즉시 응답 (DB·매핑 0)
            (revalidate 윈도우 동안 모든 요청이 캐시 재사용)
```

구성 요소별 상태:

| 요소 | 현재 | 해야 할 일 |
|---|---|---|
| `/api/revalidate` 라우트 | ✅ 존재 | `/best/*` 등 더 많은 경로/태그 커버하도록 확장 |
| 크롤러 → revalidate 호출 | 🟡 주석 처리됨([run.ts:935](../crawler/run.ts)) | 주석 해제 + 시크릿/URL 주입 |
| 일간 가격 크롤(Actions) | ✅ 운영 중 | 끝에 revalidate 트리거 추가 |
| `generateStaticParams`(/best) | ✅ 존재 | 그대로 |
| R2 인크리멘털 캐시 | ❌ dummy | **R2 버킷 + 바인딩 + open-next.config 설정** |
| KV 태그 캐시 | ❌ dummy | **KV 네임스페이스 + 바인딩**(태그 기반 revalidate용) |
| 리스트 페이지 `revalidate` | ❌ 없음 | 홈·/c·/skin 등에 `revalidate` 추가 |

## 4. 구현 단계

### P0 — 캐시 연결 + 주기 정렬 (가장 큰 효과)
1. R2 버킷 + KV 네임스페이스 프로비저닝, [wrangler.jsonc](../wrangler.jsonc)에 바인딩 추가.
2. [open-next.config.ts](../open-next.config.ts)를 R2 인크리멘털 캐시 + KV 태그 캐시 + `enableCacheInterception`으로 설정.
3. 리스트 페이지(홈·`/c`·`/c/[category]`·`/skin/...`·`/pick/[badge]/[category]`)에 `export const revalidate = 86400`(일 1회) 추가. `/best*`는 이미 있음(주기만 조정).
4. 크롤러 일간 잡 끝에 `POST /api/revalidate` 활성화([run.ts:935](../crawler/run.ts) 주석 해제) → 매일 가격 반영. 주간 import 잡에도 동일 트리거 추가.
5. `/api/revalidate`가 `/best/*`까지 무효화하도록 경로/태그 확장.

### P1 — 재생성 1회 비용도 축소 (캐시 미스/revalidate 시점)
6. 안 쓰는 `current_prices` 패치 제거(즉시·무위험).
7. `getProducts()`/`getSeoPageData()`를 `cache()`로 감싸 `/best`의 이중 매핑 제거.
8. `mapToUIProduct`의 `.find/.filter`를 사전 구성 `Map` 조회로 → O(N×L)→O(N+L).

### P2 — 클라이언트 페이로드 (선택)
9. 홈 `allProducts` 경량 projection / 리스트 서버 컴포넌트화. (무한 스크롤은 이 단계에서도 후순위)

## 5. 트레이드오프 / 리스크

- **신선도 지연:** 캐시 윈도우(일 1회) 동안은 그 시점 스냅샷이 보인다. 가격은 어차피 하루 1회 크롤이라 손해 없음. 긴급 수정은 `/api/revalidate` 수동 호출로 즉시 반영 가능.
- **revalidate 누락 시 stale:** 크롤러가 revalidate 호출에 실패하면 페이지가 오래된 채 남는다 → 안전망으로 시간 기반 `revalidate`(예: 1일)를 함께 두면 호출 실패해도 자동 만료된다(시간 기반 + 온디맨드 병행 권장).
- **R2/KV 운영 비용:** 소규모 트래픽에선 무료~수 cent 수준. 캐시로 절감되는 Worker CPU·Supabase 쿼리가 훨씬 큼.
- **무한 스크롤 도입 시 SEO 손상:** `/best/*`엔 적용 금지(크롤러가 전 상품을 못 봄). 일반 리스트도 캐시로 충분하므로 굳이 필요 없음.
- **추천 선정의 가격 의존(§2.2):** "주 동결"을 엄격히 원하면 스냅샷(§2.5) 필요. 일 1회 재계산으로 두면 단순하지만 추천 구성이 가격 변동에 따라 약간 움직일 수 있다(대개 바람직).

---

## 6. 종합 평가

- **채택:** 캐시 연결 + 일 1회 ISR + 크롤러 온디맨드 revalidate + SEO 정적화 → 제안의 의도를 **이미 있는 인프라로 최소 변경**으로 실현. "매 요청 계산"이 사라지고, 제품(주간)/가격(일간) 주기가 자연 정렬됨.
- **보류:** 무한 스크롤 — 소형 카탈로그엔 과하고 SEO에 불리. 페이로드 축소는 더 단순한 방법으로.
- **다음 결정 사항:** (a) ISR 일 1회로 갈지 vs (b) 스냅샷/풀 SSG로 "선정 주 동결"을 엄격히 구현할지. 트래픽·운영 부담 보고 a → 필요 시 b로 단계적 이행 권장.

## 7. 갱신(2026-06-30) — 가격 디커플링: 최소 부하 최종 설계 ★권장

운영자 추가 요구:
- 제품은 주기적으로 바뀌지 않는다(불규칙·드묾).
- SEO/추천 페이지가 **실시간으로 "가장 점수 높은 제품"을 전 상품 비교해 고를 필요 없다.** 일주일에 한 번 정도, 또는 정적이어도 된다.
- **가격은 리스트에서 안 보여도 된다** — 제품 링크만 걸고, 가격은 상세 페이지(클릭 후)에서 보면 된다.
- 최대한 부하 없이.

### 7.1 핵심 발견 — 가격 계산의 위치가 거꾸로다

| 페이지 | 현재 가격 계산 | 비용 | 적절성 |
|---|---|---|---|
| 리스트/SEO(`/`, `/c/*`, `/best/*` …) | **전 상품**의 풀 가격 비교(mapToUIProduct over all) | 매우 큼(O(N×L)) | ❌ 카드의 "최저가·비교" 한 줄 위해 과도 |
| 상세(`/p/[slug]`) | **그 제품 1개**만 scoped 조회([getProductDetailPageData](../lib/queries/index.ts)) | 작음 | ✅ 가격 비교가 있어야 할 곳 |

→ 즉 **"가격은 상세에서"는 이미 구현돼 있다.** 리스트가 카드 한 줄 위해 전 상품 가격을 매번 다시 계산하는 것이 낭비의 핵심.

### 7.2 권장 설계 A — 리스트/SEO는 가격 없는 정적 카탈로그 + 가격은 상세에서 (최소 부하)

- **리스트/SEO 페이지**: 카드에 이미지·브랜드·제품명·배지·`viewty_score`·상세 링크만. **가격/판매처 비교 제거.**
  - 필요한 데이터: `products`(+ `categories`, `badges`)만. **`listings`·`listing_prices_public`·`current_prices` 조회 안 함.** → 쿼리·매핑 비용 급감.
  - 제품이 드물게 바뀌므로 **긴 시간 기반 `revalidate`(예: 7일) + 제품 import 시 온디맨드 `/api/revalidate`**. 또는 import 시 풀 SSG 재빌드(완전 정적, Worker 렌더 0).
  - 추천 순위 = `viewty_score`(제품 필드, 안정적)로 정렬 → 가격 의존 제거되어 "주 1회/정적"이 자연스러움.
- **상세 페이지 `/p/[slug]`**: 지금처럼 그 제품의 판매처별 최저가 비교를 보여줌. **`revalidate = 86400`(일 1회)** + 일간 크롤러의 온디맨드 revalidate로 가격 매일 반영. scoped 조회라 1제품당 하루 1회 렌더 → ~94제품이면 무시 가능.
- **홈**: 추천 캐러셀을 `viewty_score` 기반 정적 선정 + 가격 없는 카드. `officialPicks`(정가-대비, 가격 종속)는 홈에서 빼거나 7.3 방식으로.

**부하 결과:** 리스트/SEO = 사실상 정적 에셋(핫패스에 DB 없음, 제품 바뀔 때만 재생성). 가격 계산 = 사용자가 실제로 클릭한 제품에만 발생(카탈로그 크기와 무관, 제품당 일 1회 상한). **이게 가장 부하 낮은 구조.**

**트레이드오프(정직하게):**
- 리스트 카드에서 "최저가 ₩X · 쿠팡·올영 비교"가 사라지고 카탈로그가 됨(운영자 수용함).
- SEO: `/best` 리스트의 per-product Offer JSON-LD가 빠짐 → 가격 rich-snippet은 **상세 페이지의 AggregateOffer**(이미 존재)가 담당. `/best`는 추천 ItemList + FAQ + Breadcrumb로 여전히 색인 가치 유지. 단 `/best` 카피의 "최저가 비교" 문구는 톤 조정 필요.

### 7.3 대안 B — 카드에 "최저가 숫자 1개"는 남기고 싶을 때 (current_prices 부활)

리스트에 풀 비교는 빼되 **단일 최저가 숫자**는 보여주고 싶다면, [performance-analysis.md](performance-analysis.md)에서 "삭제 대상"이던 **`current_prices` 요약 테이블이 정확히 이 용도**다(상품당 `base_lowest_price` 1행, 크롤러가 매일 갱신 중).

- 리스트 쿼리 = `products` + `current_prices`(product_id 조인, 제품당 1행 조회) → **per-listing 매핑 없이** "최저가 ₩X"만 카드에 표시.
- 풀 판매처 비교는 상세에서.
- 비용: 7.2보다 약간 높음(최저가가 매일 바뀌므로 리스트도 일 1회 revalidate 필요). 그래도 현재 풀 매핑 대비 훨씬 가벼움.
- 효과: 가격 SEO·전환율(가격 보이면 클릭률↑) 일부 유지 + 부하 낮음. **"삭제" 대신 "경량 최저가 소스로 재활용"** 으로 권고 변경 가능.

### 7.4 권고

- **기본 채택: 설계 A** — 운영자 요구("가격은 상세에서, 리스트는 정적, 최소 부하")에 가장 부합하고 부하 최저.
- **선택: 설계 B** — 카드에 최저가 숫자를 유지하고 싶으면 `current_prices`로 싸게. (이 경우 §P1의 "current_prices 삭제"는 보류.)
- 두 경우 모두 **상세 페이지가 가격의 단일 책임 지점**이 되고, 리스트/SEO는 제품이 바뀔 때만 재생성되는 (거의) 정적 페이지가 된다.
- 완전 정적 플로어를 원하면 제품 import 시 `next build`+`cf:deploy`(풀 SSG); 재배포가 부담이면 긴 `revalidate`+온디맨드로 동등 효과.

# 제목 파싱 재설계안: 정준 단위(canonical unit) 모델

> `volume-unit-conflation-analysis.md`(증상 A~E)의 해결안.
> "LLM과 규칙기반 파싱을 언제 어떤 기준으로 수행할지"와 "어떤 로직으로 title을
> 파싱해야 하는지"를 하나의 원칙으로 통합한다. 2026-07-06 설계.

---

## 0. 문제의 본질 (분석 문서 요약)

증상 A~E는 개별 버그가 아니라 **공통 설계 결함** 하나에서 나온다:

> 파이프라인의 수량 모델이 "(ml 용량, 개수)" 하나뿐이라,
> **매(시트)·개(기기) 제품도 ml 제품의 틀에 구겨 넣는다.**

- `volume_ml`은 이름부터 ml인데 매/개 제품의 "크기"까지 담는다 → 라벨만 바꿔 붙이면 "185매/200개".
- `unit_price`는 항상 `가격 ÷ ml값`(ml당)인데 표시 계층이 `volume_unit` 라벨을 붙여 "매당/개당"인 척 한다.
- 매수(80매)는 `total_quantity`(팩 개수, ≤20 clamp)에 들어갈 자리가 없어 **파싱돼도 버려진다**.
- 가드(`volumeUnit='매'→ml 용량 제거`)는 있지만 **일부 경로(normalize Check-3)만 비켜가면 무력**.

따라서 해결책도 "가드 하나 더"가 아니라 **수량 모델 자체를 단위-인지형으로 교체**해야 한다.

## 1. 핵심 원칙: 정준 단위 불변식

**모든 파싱 결과는 "그 제품의 정준 단위(= `products.volume_unit`)" 좌표계로 표현된다.**

```
ParsedQuantity {
  unit:        'ml' | 'g' | '매' | '개'   // = product.volume_unit (제품의 정준 단위)
  unitSize:    number | null              // 판매 1단위의 크기 — 정준 단위 기준
                                          //   ml/g 제품: 용량   (185ml → 185)
                                          //   매  제품: 매수   (80매입 → 80)  ← ml은 크기 아님
                                          //   개  제품: 항상 1 (기기 1대)
  packCount:   number                     // 판매 단위 묶음 수 (1개, 10개입→10, 1+1→2)
}
```

파생 규칙 (전부 이 모델에서 기계적으로 나온다):

| 정준 단위 | 제목의 ml/g 토큰 | unitSize 소스 | 단가 표시 | 크기 라벨 |
|---|---|---|---|---|
| ml/g | **크기** | 용량 토큰 | ml당/g당 = 개당가 ÷ unitSize | "185ml" |
| 매 | **에센스 함량(메타)** — 크기 아님, 폐기 | 매수 토큰(N매/장/입) 없으면 DB | 매당 = 개당가 ÷ unitSize(매수) | "80매" |
| 개 | 동봉품 용량 — 크기 아님, 폐기 | 고정 1 | 단가 라인 **비표시**(개당가=가격 그 자체) | "1개" 또는 비표시 |

이 불변식이 지켜지면 A·B·C'·E("숫자는 ml, 라벨은 매/개")는 **구조적으로 발생 불가능**해진다.
프롬프트 v6가 이미 이 규칙을 LLM에게 말하고 있다 — 이제 **코드 모델이 같은 것을 강제**해야 한다.

## 2. 규칙 vs LLM 역할 재정의

현재 게이트의 기준은 "제목이 깨끗한가(신호 구조)"다. 이를 다음으로 바꾼다:

> **규칙(regex)** — 제목이 *정준 단위 좌표계에서 결정적(deterministic)*일 때 확정한다.
> **LLM** — *구성 의미론(semantics)* 판단이 필요할 때만 호출한다:
>   세트냐 번들이냐, 증정이냐 본품이냐, 리필이 단품이냐 1+1이냐, 옵션선택이냐.

즉 "규칙=쉬운 것, LLM=어려운 것"이 아니라 **"규칙=산술, LLM=의미"**로 분업한다.
단위 좌표계를 알면 지금 LLM으로 가는 상당수가 산술 문제로 격하된다.

### 2.1 규칙 확정 범위 (게이트 확장)

기존 3개 route(단품/×N/시트)에 **단위-인지 결정 패턴**을 추가.
(각 항목의 기대 효과는 §8 실측 검증 — 운영 제목 204건 — 으로 뒷받침된다.)

- **무신호 괄호/대괄호 태그 스트립** (실측 최대 효과 — needs-llm의 41%가 이것):
  운영 제목의 괄호는 대부분 마케팅 태그다 — "[10% 추가적립]", "[흔적미백]", "(신세계 강남점)",
  "(퍼플/그린)". 현재 게이트는 "잔여 괄호 존재" 자체를 needs-llm 사유로 삼아 **명백한 단품
  45건을 LLM에 보낸다.** 개선: 괄호 세그먼트 안에 *구성 신호가 전혀 없으면*(숫자+단위 토큰 ×,
  세트/기획/증정/리필/본품/종/+ ×) 그 세그먼트를 제거하고 게이트를 계속 진행한다.
  - "[흔적미백]메디큐브 토너 250ml" → "메디큐브 토너 250ml" → trivial-single ✅
  - "(기획/단품)", "(+10ml)", "(리필)" → 신호 있음 → 유지 → LLM (안전).
- **매 제품의 표준 패턴** — `[Xml/g,] N매|장|매입|개입 [, M개]` (실측 11건 구제):
  현재는 "카운트 토큰 2개"(80매입 + 1개)라서 needs-llm으로 새는데, 매 제품에선 모호하지 않다.
  → `unitSize=N(매수), packCount=M(기본 1)`, ml 토큰은 폐기. **결정적 → 규칙 확정.**
  - "비플레인 패드 185ml, 80매입, 1개" → {unit:'매', unitSize:80, packCount:1} ✅ (증상 E)
  - "리얼베리어 마스크 27ml, 10개입, 1개" → 매 제품이면 {unit:'매', unitSize:10, packCount:1} ✅ (증상 C')
  - "아로셀 겔 마스크 42g, 4개입, 1개" → {unit:'매', unitSize:4, packCount:1} ✅ (증상 B)
  - **"본품 N매" 허용**: "메디힐 더마 토너 패드, 본품 100매, 1개"처럼 매수에 붙은 '본품'은
    세트 신호가 아니다(실측 존재) — 시트 토큰에 인접한 '본품'은 통과시킨다.
  - **제품명 '더블' 무시**: "메디필 더블 타이트 패드 70매"의 '더블'은 제품명이다(실측 존재).
    ×2 신호는 기존대로 `더블기획/더블팩/더블구성`만 인정.
  - 시트 토큰이 2개 이상이거나(90매+12매) 세트/증정 키워드 동반 → 여전히 LLM.
  - **매 제품인데 시트 토큰이 없고 ml/g만 있는 제목**("아로셀 겔 마스크 42g" — 실측 2건):
    ml은 함량이므로 폐기, unitSize는 **DB 매수로 폴백**. DB 실측(§7-b)으로 매 제품의
    volume_ml이 전부 매수(1~100)임을 확인 — **DB 폴백은 지금 즉시 안전**하다.
    (§3.4는 이 상태를 미래에도 잠그는 예방 장치.)
- **개(기기) 제품의 보수 기본값**: 카운트 토큰이 기기명에 인접하지 않으면 {unitSize:1, packCount:1}.
  ml/g·"포함/구성/+"가 있으면 무조건 LLM(기기 번들은 진짜 의미 문제, 증상 A).
  실측의 개-단위 needs-llm 2건(오큐라·듀얼소닉)은 모두 진짜 번들 → LLM행이 맞다.
- **리필 판별의 결정 케이스** (증상 D):
  - `productName`에 '리필' 포함 **또는** 제목에서 '리필'이 제품 지칭부에 붙음("쿠션 리필", "(리필)")
    이고 `본품`/`+` 페어링이 없으면 → **리필 = 제품명 일부, 단품 1개. 규칙 확정.**
  - `본품+리필` / `+리필` / `리필 증정` → 합산 2 (기존 1b 유지).
  - 그 외 모호("리필 기획" 등) → LLM.
  - (실측 204건 중 0건 — 저빈도지만 증상 D의 원인이므로 유지. 비용도 미미.)
- **소소한 토큰 커버리지 3건** (실측 각 1~2건):
  - **L/리터 단위**: "바디워시 1L, 1개"의 1L을 용량 토큰으로 인식(1L=1000ml). 현재는 미인식 →
    "1개"가 비인접 카운트로 취급돼 LLM행.
  - **`×N개` 동일 토큰**: "토너 250ml X 2개"는 배수와 카운트가 *같은 신호*다. 현재는
    "multiplier+count 동시(모호)"로 LLM행 → `[xX×*]\s*N개`를 배수 하나로 처리해 규칙 확정.
  - **역순 카운트+용량**: "샴푸 …, 1개, 530ml"처럼 카운트가 용량 *앞*에 오는 나열형(실측 1건).
    세트/증정 신호가 없고 카운트 토큰 1개 + 용량 토큰 1개뿐이면 순서와 무관하게 결정적 →
    {unitSize=530, packCount=1} 규칙 확정.

### 2.2 LLM 위임 범위 (축소·명확화)

- `+` 결합(리필 결정 케이스 제외), 세트/기획/패키지/기프트, 증정·사은품·샘플 분리 판단,
  N종(택1 vs 세트), 기기+화장품 동봉(증상 A), 이종 용량 다수, 그 외 게이트 탈락 전부.
- 프롬프트는 v6 유지 + **v7 추가 1줄**: "큐레이션 제품명(`productName`) 자체가 리필 제품이면
  제목의 '리필'은 제품명의 일부다 — main_count=1" (v5의 X2/2종 예외와 동일 패턴).
- 가드 보강: `countHasEvidence`에서 productName에 '리필'이 있으면 **'리필' 단독은 count>1의
  근거로 인정하지 않음** (titleParseGuards.ts:35 `/리필|본품|더블/` 완화).

### 2.3 판정 순서 (최종)

```
1. title_parse_cache (manual > llm[prompt_version])          … 기존 유지
2. 단위-인지 게이트: 정준 단위 좌표계에서 결정적?  → 규칙 확정
     (기존 3 route + 매-표준패턴 + 기기-보수기본 + 리필-제품명)
3. 의미 판단 필요 → LLM(v7) → titleParseGuards(기존 + 리필근거 완화)
4. LLM 불가/기각 → regexFallback + needsInspection            … 기존 유지
5. 모든 경로의 출력이 ParsedQuantity(정준 단위)로 정규화되어야만 normalize에 진입
```

**5가 새 원칙의 관문이다**: 어떤 경로로 파싱했든 정준 단위로 변환하는 단일 함수
`toCanonicalQuantity(ext, ctx)`를 통과해야 normalize가 소비한다. 가드 우회가 불가능해진다.

## 3. 파이프라인 배관 수정 (누수 3곳 봉쇄)

### 3.1 normalize Check-3(무가드 옛 regex) 삭제 — 항상 parsedPackage 주입

현재 `LLM_TITLE_PARSE=off`이거나 저신뢰면 normalize가 `extractPackageFromTitle`을 직접 불러
**가드 없는 경로**가 생긴다(분석 §2.4-2). 해결:

- `run.ts`에서 **LLM off여도 항상** `parsePackage(title, ctx /* llmExtract 미주입 */)`를 호출해
  `offer.parsedPackage`를 주입한다. llmExtract가 없으면 게이트+regexFallback만 도는데,
  이 경로도 단위 정합 가드를 통과한다(추가 비용 0 — LLM 호출 없음).
- `normalize.ts`의 Check-3(직접 regex 재파싱, 204-213)과 개수 블록의 `extractPackageFromTitle`
  폴백(141)을 **삭제**. normalize는 `offer.parsedPackage`(정준화된 결과)만 소비.
- 효과: regex 엔진 이원화 해소(stage-3 방향과 일치). `extractPackageFromTitle`은
  parsePackage 내부 폴백으로만 남는다.

### 3.2 매수/크기를 스키마에 보존 (분석 P1)

- `NormalizedPrice`에 `unit_size`(정준 단위 기준 1단위 크기) 추가.
  `total_ml` → 의미를 "정준 단위 총량"으로 재정의(매 제품 = 총 매수). 컬럼명은 유지(마이그레이션 최소화), 주석으로 명시.
- clamp 분리: **packCount 1~20**(팩 수) / **unitSize: ml 1~2000, 매 1~300**.
  현재 `uCount≤20`이 매수 80을 죽이는 버그(분석 §2.5)가 자연 해소된다.
- `unit_price = effective_unit_price ÷ unit_size` — 매 제품이면 **진짜 매당**, ml이면 ml당.
  개(기기) 제품은 `unit_price = null`(단가 라인 비표시).

### 3.3 표시 계층: 숫자와 라벨의 출처 통일

- `lib/queries/index.ts`: `volumeMl`(크기값)과 `volumeUnit`(라벨)을 **같은 소스**(정준화된
  unit_size/unit)에서만 만든다. `StorePriceCard`는 받은 쌍을 그대로 출력 — 조합 로직 금지.
- 방어선: `volume_unit`이 매/개인데 unit_size가 ml량으로 의심되는 경우(예: 매 제품에
  unit_size>300) → 크기·단가 **비표시**(틀린 숫자보다 없는 숫자가 낫다).

### 3.4 DB 위생 — 실측 결과: 이미 깨끗 (검증은 "잠금"용으로 유지)

**2026-07-07 DB 값 전수 확인(§7-b)으로 분석 문서 §2.4-(1)의 "DB 오염" 가설은 기각됐다.**
개-단위 7개 제품 전부 `volume_ml=1`, 매-단위 11개 제품 전부 매수(1·4·60·70·90·100).
ml값(185/200/42/27)이 들어간 행은 **0건** — 1회 감사는 사실상 완료, 교정 대상 없음.

따라서 §3.4의 역할은 "교정"이 아니라 **현 상태를 불변식으로 잠그는 예방**이다:
- `sheets:import` 검증 추가: `volume_unit='매'`→`volume_ml`은 1~300 정수(매수),
  `'개'`→1(또는 소수 개수), `'ml/g'`→1~2000. 위반 행은 import 거부 + 시트에 사유 표기.
- 시사점: 증상 A~E의 잘못된 숫자는 **전부 파싱 경로에서 유입**된 것(§3.1·§3.2가 근본 해결책)
  — PR-1·PR-2의 우선순위가 실측으로 재확인됐다.

### 3.5 관측성 — 제목별 판정 트레이스 (2026-07-08 운영자 요청)

"왜 이 숫자가 됐나"를 파싱 시점에 남겨 디버깅을 한 번의 로그 검색으로 끝낸다:

- `ParsePackageResult`에 **`trace` 필드** 추가: 게이트 route/탈락 사유(reason), 적용된
  전처리(bracket-strip·L→ml·×N개·리필-제품명), LLM 사용 시 confidence·evidence·가드 적용
  내역(용량 환각 null 처리, 단위 정합 교정 등), 캐시 히트 여부(manual/llm/miss)를 담는다.
- `run.ts`가 **제목당 1줄 구조화 로그** 출력:
  `[TitleParse] product=<key> seller=<slug> src=rule|llm|cache-manual|cache-llm|fallback route=매-패턴 size=80매 pack=1 evidence="80매입" title="비플레인 …"`
  → GitHub Actions 크롤 로그에 영구 보존되어 grep으로 즉시 추적. mock/test에서도 동일 출력.
- LLM 결과는 기존대로 `title_parse_cache.result_json`에 영속(trace 포함) — 시점별 판정
  근거가 DB에도 남는다. 게이트 확정분은 로그로만(재계산 공짜라 DB 저장 불필요).
- 캐시 보강(백로그): DB 캐시 키가 제목 sha256뿐이라 제품의 DB 용량/단위 변경 시 옛 ctx로
  나온 LLM 결과가 재사용될 수 있음 → 캐시 행에 ctx(정준 단위·용량)를 저장하고 불일치 시
  miss 처리. (당장은 프롬프트 버전 bump로 일괄 무효화 가능.)

### 3.6 identity 가드 — 용량 10× 불일치 → inspection (2026-07-08 운영자 확정)

§7-c에서 발견한 "파싱은 정확하지만 다른 제품이 매칭된" 케이스(토리든 40ml 세럼 ↔
클렌징워터 400ml)의 보강. **운영자 결정: 도입.**

- 규칙: 파싱된 판매처 용량(unitSize, 정준 단위 동일할 때)이 DB 정준 용량 대비
  **10배 이상 또는 1/10 이하**면 가격을 노출하지 않고 **inspection 라우팅**(예측값 prefill
  + 사유 "용량 10× 불일치 — 다른 제품/변형 의심").
- **앵커 오퍼는 예외**: 앵커 = 운영자가 검토해 넣은 SKU이므로 10× 불일치는 오매칭이 아니라
  **DB 용량 오류**를 의미 → 노출은 유지하되 Discord 경고로 통지(운영자가 DB 교정).
- per-retailer 용량 정책(불일치는 정보성)은 10× 미만 범위에서 그대로 유지 —
  이 가드는 "다른 사이즈 변형"(2~3×)이 아니라 "다른 제품"(10×+)만 걸러낸다.
- 매 제품 폼명 불일치(마스크 제품 ↔ "크림 라이트" 오퍼) 감지는 백로그 유지(§7-c-1ⓑ).

## 5. 보상 통제: 앵커 오퍼의 불확실 파스

앵커된 SKU는 검수를 건너뛰고 노출한다(운영자 확정 SKU — 의도된 정책). 그런데 파스가
`needsInspection`인 채 노출되면 **틀린 개수/용량이 그대로 화면에 갈 수 있다**. 보상책:

> 앵커 오퍼 + 파스 저신뢰 → 그 파스의 숫자를 쓰지 말고 **보수 기본값**(packCount=1,
> unitSize=DB)으로 노출한다. "노출은 하되, 불확실한 산술은 반영하지 않는다."

normalize가 `parsedPackage.needsInspection===true`면 수량/크기 블록을 스킵(기본값 유지)
하는 한 줄 게이트로 구현 가능 — 현재도 `confidence==='high'`만 쓰므로 준수하고 있는지
확인만 하면 된다(regexFallback이 low로 떨어지는 것 재확인).

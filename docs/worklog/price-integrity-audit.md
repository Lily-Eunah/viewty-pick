## 0. 한 줄 결론

> **가격을 신뢰할 수 없는 근본 원인은 "수집 단계의 SKU 매칭 오류"다.** Naver Shopping API 매칭이 큐레이션한 **단품 SKU가 아니라, 검색 상위에 뜬 선물세트·기획·더블팩·2종세트·샘플 동봉 상품**을 그대로 가져온다. 정규화(`packageExtractor`)는 그 세트의 구성을 다시 오해석해 개당가/ml당가를 틀리게 만든다. 표시 단계의 "어제보다 N원·N% 인하·별점·리뷰수"는 **전부 mock(자산 없음)** 이라 가격 신뢰도를 더 떨어뜨린다.

## 2. 공통 근본원인 (우선순위)

### RC-A. Naver 매칭이 큐레이션 SKU가 아닌 "상위 노출 세트/기획"을 집는다 — **최우선**
- `crawler/adapters/naver.ts: pickOfficialOffer()` 는 **공식몰 + 제목 토큰 유사도 ≥ 0.5** 를 통과하는 **Naver 관련도(relevance) 최상위** offer를 그대로 채택한다.
- 결함:
  1. **큐레이션 listing의 productId를 버린다.** listing.url에 `brand.naver.com/lancome/products/10791745136` 같은 정확한 SKU가 있는데도, 매칭은 brand+name으로 **재검색**해 다른 productId(13278074879)를 가져온다. (쿠팡 어댑터는 URL productId로 앵커링하는데, Naver/OY는 안 함.)
  2. **0.5 토큰 유사도는 세트가 쉽게 통과.** 세트 제목은 제품명을 다 포함하므로 "21ml+100ml 세트", "2종 세트", "선물세트", "더블 기획" 모두 통과.
  3. **단품/세트/팩 선호 규칙이 없다.** 유세린의 경우 단품 60,900(pid 88900077622)이 결과에 **존재**하는데도, 같은 공식몰의 상위 "더블팩/2종세트"를 먼저 집는다.
  4. **용량 불일치가 reject가 아니다**(§1b 타협). DB volume이 unverified라는 이유로 50ml↔세트, 30ml↔2종세트가 통과한다.

### RC-B. `packageExtractor` 가 세트 구성을 오해석한다
- `crawler/core/packageExtractor.ts` 는 **동일 단위 부피/개수만** 모델링. 제목의 **첫 번째** 부피/개수 토큰을 집어:
  - 증정 샘플("7ml*2")을 묶음 수량으로 → 유세린 OY 개당 19,950.
  - "1매"(필름 마스크)를 수량/프로모로 → 바이오힐보 "1매)".
  - "더블/1+1/리필" 키워드 미인식 → 토리든 OY 더블 38,000·라하 OY 1+1 24,000을 단품 취급.
- **이종 구성 세트(토너+세럼, 세럼+디바이스)는 개당가·ml당가 산정이 본질적으로 불가** → 비교 제외 대상이어야 하나, 현재는 단품처럼 base 가격으로 노출.

### RC-C. 올리브영 수집 공백 (`retailer_allowlist` 0건)
- allowlist가 **완전히 비어 있어** 모든 브랜드가 "mallName에 브랜드명 포함" fallback에만 의존.
- OY는 Naver상 mallName="올리브영"이라 브랜드명을 포함하지 않음 → 랑콤 등에서 **무조건 no_offer**(랑콤 OY 미수집의 직접 원인).
- 매칭되는 경우도 OY는 Naver에 **기획/세트만** 노출되는 경향(토리든 더블 38,000) → 단품 OY 가격은 사실상 미수집.

### RC-D. 신선도/관측 공백
- `listings.last_crawled_at` = **171건 전부 NULL**. 스냅샷은 쌓이는데(`run.ts`는 fail_count·is_active·latest_matched_url·viewty_score만 업데이트) `last_crawled_at`을 **한 번도 기록하지 않음**.
- `crawl_runs` = **0건**, `crawl_errors` = **0건**. `run.ts`의 `crawl_runs.insert({ started_at: startTime ... })` 는 `startTime`이 **JS epoch 숫자**(TIMESTAMPTZ 컬럼) → 기록 실패/누락 의심. 일별 실행 이력·매칭 실패 분포를 DB에서 볼 수 없음.
- 최신 스냅샷 status 분포(최근 2000건): `ok 353 / warning 68 / no_offer 24`.

## 4. 정규화 / 정렬 정책 — 실제 동작 vs 의도

- **2트랙 단가**: 라하 토너 쿠팡 6개에서 `base 67,800 / eff(개당) 11,300 / unit(ml당) 42.6` 계산은 **정확**. 토너 케이스는 packageExtractor가 "265ml, 6개"를 제대로 잡음. 문제는 **세트 제목**(증정/리필/더블/이종구성)에서 깨진다(RC-B).
- **이종 구성 세트(토너+세럼)**: `packageExtractor`/`normalize`에 **처리 경로 없음**. 첫 부피 토큰만 잡아 단품처럼 base 가격을 노출 → **비교 제외 처리 필요**(현재 미처리).
- **최저가 정렬 기준**: 리스트 "최저가" 및 store 정렬은 **base_unit_price** 기준(`lib/queries/index.ts:127,146`). 따라서 라하 토너에서 쿠팡(팩 67,800)은 상위로 안 오고 네이버(16,800)가 최저 — **base 기준으론 의도대로**. 단 **ml당 최저는 쿠팡(42.6)** 인데 이 신호는 화면 주축에 안 쓰여, "개당/ml당이 더 싼데 비싸 보이는" 혼선 발생.
- **null(무가격) 상품 정렬**: 무가격 상품은 `lowestPrice = 0`이 되어 **"최저가순"에서 맨 앞**으로 정렬됨(`CategoryProductList.tsx:34`, `lib/queries/index.ts:303`). 의도(뒤로 보내기)와 **반대**. 화면엔 `won(0)`="가격 확인 중"으로 뜨지만 정렬상 최저가 1위. **버그.**

## 5. 추가 발견 (구조)

1. **매칭이 큐레이션 productId를 무시**: Naver/OY 어댑터는 listing.url의 productId를 쓰지 않고 재검색한다. 쿠팡 어댑터만 URL productId 앵커링. → Naver/OY도 앵커링 도입이 RC-A의 핵심 해법.
2. **`current_prices`와 `listing_prices_public` 불일치**: `current_prices`는 `run.ts`에서 warning+uprel=false 행(유세린 OY 39,900 "7ml*2")까지 포함해 base_lowest를 계산하지만, 공개 뷰(0010)는 status='ok' & latest만 노출 → **두 값이 어긋남**(유세린: current_prices 39,900 vs 뷰 105,600).
3. **`current_prices`는 웹 UI에서 死코드**: `mapToUIProduct`는 `dbPrices`(current_prices)를 인자로 받지만 **사용하지 않고** `listing_prices_public`에서 재계산한다. 표시값은 전적으로 공개 뷰 기반.

## 6. 수정 범위 제안 (우선순위)

### (A) 크롤/정규화/매칭 — **P0 (가격 신뢰도의 핵심)**
- **A1 [P0] 큐레이션 productId 앵커링**: Naver/OY 매칭에서 listing.url의 Naver productId를 1순위 앵커로 사용(쿠팡 방식). 정확 일치 시 그 offer의 가격만 채택.
- **A2 [P0] 단품/세트 판별 + 세트 비교 제외**: 제목에 `세트|기획|선물|더블|리필|2종|증정|+`/이종 단위가 있으면 "세트"로 분류해 **개당가 비교에서 제외**(또는 base만, "세트가" 라벨). packageExtractor에 set/heterogeneous 분기 추가.
- **A3 [P0] 매칭 후보 선호 규칙**: 같은 공식몰에 단품과 세트가 모두 있으면 **단품 우선**(volume==DB volume, q==1, 세트 키워드 없음) 선택. 0.5 임계만으론 부족.
- **A4 [P1] retailer_allowlist 채우기**: 브랜드별 공식 store_name 등록(올영=올리브영 mallName, 브랜드 공식몰명). RC-C 직접 해소. 등록 전엔 OY no_offer를 "데이터 공백"으로 명시.
- **A5 [P1] 용량 검증**: "200g vs 200ml" 등 단위/용량 mismatch를 inspection 큐로(현재 unit_price만 null 처리하고 가격은 통과).

### (B) mock 제거 + 뱃지 정책 — **P1**
- **B1** 별점/리뷰수(TodayDealSection) **제거**.
- **B2** previousPrice / "어제보다 N원" / "N% 인하" **제거** (가격 히스토리 테이블 생길 때까지).
- **B3** "오늘 가격 좋은 제품"·"가격하락순"을 실 인하 데이터 기반으로 교체하거나 임시 비표시(현재는 고가 우선 정렬).
- **B4** 추천 사유/뱃지 fallback 임의 부여 비표시 검토.

### (C) 표시 · 정렬 · placeholder — **P1~P2**
- **C1 [P1] null 가격 정렬 수정**: "최저가순"에서 무가격(lowestPrice=0) 상품을 **맨 뒤**로. (정렬 키에서 0/누락을 +Infinity 처리.)
- **C2 [P2] 다중팩 표시**: 팩 상품은 base(팩가)와 함께 **개당가/ml당가**를 동등 비중으로 노출(라하 쿠팡 67,800 → "개당 11,300 · ml당 42.6").
- **C3 [P2] current_prices 정리**: 死코드면 제거하거나, 공개 뷰와 동일 규칙으로 재계산해 일관성 확보.

### (D) 관측/신선도 — **P2**
- **D1** `run.ts`가 listing별 `last_crawled_at` 기록.
